# IN

O operador `IN` é usado para verificar se um valor está presente em uma lista de valores especificados. É uma alternativa mais legível e eficiente ao usar múltiplos operadores `OR`.

## Sintaxe Básica

```sql
SELECT column1, column2, ...
FROM table_name
WHERE column_name IN (value1, value2, value3, ...);
```

## Como Funciona - Passo a Passo

### Passo 1: O operador IN avalia a condição

O operador `IN` verifica se o valor da coluna está presente na lista de valores fornecida. Se estiver, a linha é retornada.

### Passo 2: Equivalência com OR

Internamente, `IN (value1, value2, value3)` é equivalente a:

```sql
column_name = value1 OR column_name = value2 OR column_name = value3
```

Porém, `IN` é mais legível e geralmente mais eficiente.

## Exemplos Práticos

### Exemplo 1: Lista de valores literais

```sql
-- Encontrar clientes de cidades específicas
SELECT id, nome, cidade
FROM clientes
WHERE cidade IN ('São Paulo', 'Rio de Janeiro', 'Belo Horizonte');
```

**Explicação detalhada:**

1. O banco de dados verifica cada linha da tabela `clientes`
2. Para cada linha, verifica se o valor da coluna `cidade` está na lista
3. Se estiver, a linha é incluída no resultado
4. Se não estiver, a linha é excluída

### Exemplo 2: IN com subquery

```sql
-- Encontrar produtos que foram vendidos
SELECT id, nome, preco
FROM produtos
WHERE id IN (SELECT produto_id FROM vendas);
```

**Explicação detalhada:**

1. A subquery `(SELECT produto_id FROM vendas)` é executada primeiro
2. Ela retorna uma lista de todos os `produto_id presentes na tabela vendas`
3. O operador `IN` verifica se cada `produto.id` está nessa lista
4. Se estiver, o produto é incluído no resultado

### Exemplo 3: IN com múltiplas colunas (alguns bancos)

```sql
-- PostgreSQL: IN com múltiplas colunas
SELECT *
FROM produtos
WHERE (categoria_id, preco) IN (
    SELECT categoria_id, AVG(preco)
    FROM produtos
    GROUP BY categoria_id
);
```

## Comportamento com NULL

### Cenário 1: Lista contendo NULL

```sql
SELECT nome
FROM clientes
WHERE cidade IN ('São Paulo', NULL, 'Rio de Janeiro');
```

**Comportamento:**

- Se `cidade` for 'São Paulo' → retorna TRUE
- Se `cidade` for 'Rio de Janeiro' → retorna TRUE
- Se `cidade` for NULL → retorna UNKNOWN (não FALSE)
- Se `cidade` for 'Curitiba' → retorna FALSE

**Resultado:** Linhas com NULL na lista não causam erro, mas também não retornam linhas com NULL na coluna.

### Cenário 2: Coluna contendo NULL

```sql
SELECT nome
FROM clientes
WHERE cidade IN ('São Paulo', 'Rio de Janeiro');
```

**Comportamento:**

- Se `cidade` for NULL → retorna UNKNOWN (não FALSE)
- A linha não será retornada

**Resultado:** Linhas com NULL na coluna não são retornadas.

## Pros e Contras

### Pros

1. **Legibilidade**: `IN` é muito mais legível que múltiplos `OR`

   ```sql
   -- Mais legível
   WHERE cidade IN ('SP', 'RJ', 'MG')
   
   -- Menos legível
   WHERE cidade = 'SP' OR cidade = 'RJ' OR cidade = 'MG'
   ```

2. **Performance**: Otimizadores de SQL podem tratar `IN` de forma mais eficiente que múltiplos `OR`

3. **Manutenção**: Adicionar ou remover valores é mais fácil

   ```sql
   -- Fácil adicionar mais valores
   WHERE cidade IN ('SP', 'RJ', 'MG', 'RS', 'PR')
   ```

4. **Subqueries**: Permite usar resultados de subqueries diretamente

   ```sql
   WHERE id IN (SELECT produto_id FROM vendas)
   ```

### Contras

1. **NULL handling**: Comportamento com NULL pode ser confuso

   ```sql
   -- Se a subquery retornar NULL, o resultado pode ser inesperado
   WHERE id IN (SELECT id FROM tabela WHERE condicao)
   ```

2. **Performance com listas grandes**: Listas muito grandes podem ter performance ruim

   ```sql
   -- Pode ser lento se a lista tiver milhares de valores
   WHERE id IN (1, 2, 3, ..., 10000)
   ```

3. **Limitações de tamanho**: Alguns bancos têm limites no tamanho da lista

## Cenários a Considerar

### Cenário 1: Lista pequena (até ~100 valores)

**Recomendação:** Usar `IN` com lista literal

```sql
WHERE id IN (1, 2, 3, 4, 5)
```

### Cenário 2: Lista grande (100+ valores)

**Recomendação:** Usar `IN` com subquery ou tabela temporária

```sql
-- Opção 1: Subquery
WHERE id IN (SELECT id FROM tabela_temporaria)

-- Opção 2: JOIN
SELECT t1.*
FROM tabela1 t1
JOIN tabela_temporaria t2 ON t1.id = t2.id
```

### Cenário 3: Valores dinâmicos (aplicação)

**Recomendação:** Usar parâmetros para evitar SQL injection

```java
// Java/JDBC
String sql = "SELECT * FROM clientes WHERE id IN (?, ?, ?, ?)";
PreparedStatement stmt = connection.prepareStatement(sql);
stmt.setInt(1, 1);
stmt.setInt(2, 2);
stmt.setInt(3, 3);
```

### Cenário 4: Subquery retornando muitos valores

**Recomendação:** Verificar se EXISTS seria mais eficiente

```sql
-- IN (pode ser lento se subquery retorna muitos valores)
WHERE id IN (SELECT produto_id FROM vendas)

-- EXISTS (geralmente mais eficiente)
WHERE EXISTS (SELECT 1 FROM vendas WHERE vendas.produto_id = produtos.id)
```

## IN vs Alternativas

### IN vs OR

```sql
-- IN (recomendado)
WHERE cidade IN ('SP', 'RJ', 'MG')

-- OR (menos legível)
WHERE cidade = 'SP' OR cidade = 'RJ' OR cidade = 'MG'
```

**Escolha:** `IN` é preferível por legibilidade.

### IN vs JOIN

```sql
-- IN
SELECT * FROM produtos
WHERE id IN (SELECT produto_id FROM vendas)

-- JOIN (geralmente mais eficiente para grandes volumes)
SELECT DISTINCT p.*
FROM produtos p
JOIN vendas v ON p.id = v.produto_id
```

**Escolha:** `JOIN` para grandes volumes, `IN` para pequenos volumes ou quando legibilidade é prioritária.

### IN vs EXISTS

```sql
-- IN
SELECT * FROM produtos
WHERE id IN (SELECT produto_id FROM vendas)

-- EXISTS (geralmente mais eficiente)
SELECT * FROM produtos p
WHERE EXISTS (SELECT 1 FROM vendas v WHERE v.produto_id = p.id)
```

**Escolha:** `EXISTS` é geralmente mais eficiente quando a subquery retorna muitos valores.

## Dicas de Performance

1. **Índices**: Certifique-se de que a coluna usada no `IN` tem índice

   ```sql
   CREATE INDEX idx_clientes_cidade ON clientes(cidade);
   ```

2. **Evite NOT IN com NULL**: `NOT IN` pode ter problemas de performance com NULL

   ```sql
   -- Pode ser lento se subquery retorna NULL
   WHERE id NOT IN (SELECT id FROM tabela)
   
   -- Melhor alternativa
   WHERE NOT EXISTS (SELECT 1 FROM tabela WHERE tabela.id = produtos.id)
   ```

3. **Use EXISTS para subqueries complexas**: `EXISTS` geralmente é mais eficiente

   ```sql
   WHERE EXISTS (SELECT 1 FROM vendas WHERE vendas.produto_id = produtos.id)
   ```

## Exemplos Avançados

### Exemplo 1: IN com CASE

```sql
SELECT 
    nome,
    CASE 
        WHEN regiao IN ('Norte', 'Nordeste') THEN 'Norte/Nordeste'
        WHEN regiao IN ('Sul', 'Sudeste') THEN 'Sul/Sudeste'
        ELSE 'Centro-Oeste'
    END as regiao_grupo
FROM estados;
```

### Exemplo 2: IN com agregação

```sql
SELECT 
    categoria_id,
    COUNT(*) as total
FROM produtos
WHERE id IN (SELECT produto_id FROM vendas WHERE data_venda >= '2024-01-01')
GROUP BY categoria_id;
```

### Exemplo 3: IN múltiplos (aninhado)

```sql
SELECT *
FROM clientes
WHERE cidade IN (
    SELECT cidade 
    FROM filiais 
    WHERE regiao IN ('Sudeste', 'Sul')
);
```

## Resumo

- **Use IN quando**: Lista pequena, legibilidade é importante, valores conhecidos
- **Evite IN quando**: Lista muito grande, subquery retorna muitos valores, NULL é um problema
- **Alternativas**: EXISTS para subqueries, JOIN para grandes volumes, OR para casos simples
- **NULL**: Lembre-se que NULL no resultado de IN retorna UNKNOWN, não FALSE
